Core Features

Encryption Engine

  • ▹ Cipher: AES-256-GCM (NIST-approved AEAD)
  • ▹ Auth: 128-bit GCM tag untuk integritas data
  • ▹ Key Size: 256-bit master key + PBKDF derived keys
  • ▹ IV: 96-bit random per-file

Key Derivation

  • ▹ Algorithm: Argon2id (Memory-hard, timing-resistant)
  • ▹ Time Cost: 5 iterations untuk mencegah brute-force
  • ▹ Memory: 256 MB per op mencegah rainbow tables
  • ▹ Salt: 256-bit random, unik per master key

Encryption Flow

1

Startup & Anti-Debug

Saat mesin BitLocker dieksekusi oleh User, aplikasi pertama kali berjalan di latar belakang tanpa jendela (*stealth boot*). Sistem memindai proses OS untuk mendeteksi *Debugger* atau alat forensik RAM (memanggil API `IsDebuggerPresent`). Jika aman, ia menggandakan dirinya sebagai berkas acak (Hash Morphing) ke folder `%TEMP%` untuk menyembunyikan identitas aslinya.

2

Key Generation & Exfiltration

Sistem menciptakan 512-bit Master Key primer. Kunci primer ini disandikan ulang menggunakan RSA *Public Key* (sehingga hanya bisa dibuka oleh kunci privat dari pembuat *ransomware*). Selanjutnya, sebuah *Send Thread* asinkron mulai berjalan mencoba mengirim device_id dan encrypted_key ke panel kontrol (Vercel API Anda) lewat lalu-lintas HTTPS multipart atau JSON secara terus-menerus hingga berhasil (*Retry Loop*).

3

Cryptography execution

GUI horor menyerupai "Terminal Retas" muncul, mengunci akses *Foreground* PC. Di latar belakang, *engine* merayapi direktori (*recursive target*) untuk mencari dokumen eksekutif (docx, xls, pdf, txt, dsb). Tiap berkas dienkripsi bit-demi-bit dengan cipher AES-256-GCM dan digabungkan bersama 96-bit Initialization Vector (IV) agar tiap berkas unik. Ekstensi awal diganti menjadi .locked, dan berkas aslinya ditimpa (*overwrite 3x pass gutmann*) sehingga tidak mungkin dibalikkan menggunakan piranti kembalikan berkas biasa.

Cleanup & Self-Destruct

Setelah target tuntas dienkripsi, jejak kunci *plaintext* dari memori di RAM dibersihkan paksa lewat instruksi OPENSSL_cleanse() (menjamin ketahanan dari *Cold Boot Attack*). Proses lalu akan menghapus log operasinya dari SSD dan menghancurkan programnya sendiri secara otomatis (*secure suicide*), menyisakan hanya berkas .locked dan instruksi penembusan tebusan untuk User yang terdampak.

Security & Anti-Forensics

Secure File Deletion

Menggunakan metode Gutmann-style (3-pass overwrite). Sistem menimpa data asli dengan byte acak, invert data, lalu byte acak lagi sebelum memutus handle berkas. Ini membuat pemulihan oleh peranti lunak forensik disk seperti Recuva atau EaseUS menjadi mustahil.

Memory Sanitization

Pemanfaatan instruksi OPENSSL_cleanse() diterapkan seketika pada string password, buffer *plaintext*, serta materi kunci agar mencegah serangan *Cold Boot RAM Forensics*.

Zero-Knowledge Recovery

Client menghasilkan 512-bit random key. Kunci primer ditransmisikan dalam enkapsulasi RSA terenkripsi. Server hanya menyimpan sesi ID, dan aplikasi klien *self-destruct* kuncinya usai digunakan. Admin backend tidak dapat merekonstruksi data tanpa persetujuan alat klien lokal.

The Phantom-7 Protocol

BitLocker Security Suite mahakarya Native C++ Security tanpa dependensi eksternal, dikompresi menjadi binari ultra portable ~3MB via UPX. Menyajikan efek psikologis "Total Lockdown".

Psychological Warfare GUI

GUI dibangun dengan Double-Buffered GDI+. Mengambil fokus paksa (Foreground Locking), melumpuhkan navigasi standar OS, serta memberikan umpan balik mesin tik ala Terminal Intelijen per ketukan tombol.

Active Self-Defense

Secara dinamis memodifikasi Access Control List (DACL) pada layanannya sendiri untuk menolak perintah PROCESS_TERMINATE dan menyegel eksekusi Thread GUI.
